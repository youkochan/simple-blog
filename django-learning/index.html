django学习记录

May 10, 2016

Reading time ~5 minutes

真的好久没写过东西了，最近打算在入职之前找点东西做，于是扯上杨大大准备一起写点小东西。需要用到 Django，之前寒假自己折腾 Django 的时候更多时候把注意力放在了快速开发上面，对于一些比较基础的知识并没有很深入地去了解。特此记录。

检测 Django 版本

$ django-admin.py --version
1.9.1

创建一个 Django 项目

可以使用如下指令来创建一个 Django 项目：

$ django-admin.py startproject ss_panel

这个指令会在当前目录下生成一个 ss_panel 目录，目录结构如下：

$ tree ss_panel
ss_panel
├── manage.py
└── ss_panel
    ├── __init__.py
    ├── settings.py
    ├── urls.py
    └── wsgi.py

1 directory, 5 files

下面来具体解释一下这些文件以及文件夹的意义：

  • 外层的 ss_panel：这个仅仅只是一个项目的容器，你可以取任何一个你喜欢的名字，默认的名字就是生成项目时候输入的名字。
  • manager.py：一个命令行工具，可以用来与 Django 项目进行交互。参考 链接
  • 内层的 ss_panel：内层的 ss_panel/ 目录是你的项目的真正的Python包
  • ss_panel/__init__.py：一个空文件，用于告诉 python 这个目录应该被看做一个包~
  • ss_panel/setting.py：该项目的一些设置
  • ss_panel/urls.py：该项目的路由 URL 声明
  • ss_panel/wsgi.py：用于你的项目的与WSGI兼容的Web服务器入口。参考 链接

数据库

讲道理的话，web 项目不能没有数据库啊（一般来说），而 Django 对各种数据库都会有很好的支持。默认情况下，Django 使用的是 SQLite。SQLite 是一种轻量级的数据库，不需要额外的环境配置和软件安装，SQLite 对于数据库的操作其实就是对一个数据库文件的操作，而操作方法都被封装在了对应的 python 包之中。对于 ss_panel 的需求来说，SQLite 就已经足够了。

下面是 setting.py 文件中有关数据库文件的那一段：

# Database
# https://docs.djangoproject.com/en/1.9/ref/settings/#databases

DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.sqlite3',
        'NAME': os.path.join(BASE_DIR, 'db.sqlite3'),
    }
}
  • ENGINE：默认的数据库引擎类型，django 支持 ‘django.db.backends.sqlite3’, ‘django.db.backends.postgresql_psycopg2’, ‘django.db.backends.mysql’或’django.db.backends.oracle’
  • NAME：数据库的名称。如果你使用 SQLite ，数据库将是你计算机上的一个文件，如果是这样的话，NAME应该是这个文件的绝对路径，包括文件名。默认值是 os.path.join(BASE_DIR, ‘db.sqlite3’)，它将文件保存在你项目的目录中。

已安装的应用

下面是 setting.py 文件中有关已安装的应用的那一段：

# Application definition

INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
]

它保存这个 Django 实例中激活的所有的 Django 应用的名字。 应用可以使用在多个项目中，而且你可以将这些应用打包和分发给其他人在他们的项目中使用。

默认情况下，INSTALLED_APPS 包含下面的应用：

  • django.contrib.admin —— 管理站点。
  • django.contrib.auth —— 认证系统。
  • django.contrib.contenttypes —— 用于内容类型的框架。
  • django.contrib.sessions —— 会话框架。
  • django.contrib.messages —— 消息框架。
  • django.contrib.staticfiles —— 管理静态文件的框架。

上面的部分应用至少需要使用一个数据库表，因此我们需要在使用它们之前先在数据库中创建相应的表。要做到这一点，请运行以下命令：

$ python manage.py migrate

migrate 查看 INSTALLED_APPS 设置并根据 mysite/settings.py 文件中的数据库设置创建任何必要的数据库表，数据库的迁移还会跟踪应用的变化

创建应用

  • 应用：应用是一个Web应用程序，它完成具体的事项 —— 比如一个博客系统、一个存储公共档案的数据库或者一个简单的投票应用
  • 项目：项目是一个特定网站中相关配置和应用的集合。一个项目可以包含多个应用。一个应用可以运用到多个项目中

创建应用：

./manager.py startapp _your_app_name_

由于 ss_panel 是一个小型的项目，木有那么多需要新建的应用，主要应用分为两个：1，面板部分的操作；2，定时任务，用于获取某个端口的流量。

创建模型

新建一个 app 后，在 models.py 中就可以定义自己的模型了，通过 ORM 机制，对于模型类实例的操作将会转换为对于数据库的操作，这样就可以省下了不少编码工作。

from django.db import models
from django.utils import timezone


class Client(models.Model):
    user = models.ForeignKey('auth.User')
    port = models.IntegerField()
    password = models.CharField(max_length=50)
    usage_sum = models.BigIntegerField()
    usage_daily = models.BigIntegerField()
    usage_monthly = models.BigIntegerField()
    account_balance = models.IntegerField()
    created_date = models.DateTimeField(default=timezone.now)

    def __str__(self):
        return "port:%s user:%s" % (self.port, self.user)

其中，Client 可以看做一个数据表的名称，其中的每一个成员为数据表的一个字段。

  • class 是一个特殊的关键字，表明我们在定义一个对象
  • Client 是我们模型的一个名字
  • models.Model 表明 Client 是一个Django模型，所以 Django 知道它应该被保存在数据库中
  • models.CharField - 这是你如何用位数有限的字符来定义一个文本。
  • models.TextField - 这是没有长度限制的长文本。
  • models.DateTimeField - 这是日期和时间。
  • models.ForeignKey - 这是指向另一个模型的连接。
  • models.BigIntegerField - 这是大整数的类型。
  • models.IntegerField - 这是整数的类型。
  • __str__ - 函数的定义影响着对象在 admin 中的显示效果。

使用命令行形式进行数据库管理

执行 ./manager.py shell 会自动启动带有环境变量的 python 解释器。直接使用 python 执行会报错。

import django
django.setup()                      # 不使用 ./manager.py shell 会报错
from panel.models import Client     # 自定义的 model

Client.objects.all()
# [<Client: port:60001 user:test_user1>, <Client: port:60002 user:test_user2>]

client = Client.objects.all()[0]
# the first client

client.port = 9999
client.save()
# <Client: port:9999 user:test_user1>

urls

urls 表示访问某一个网址时执行哪一个操作，可以通过一下语句把访问我们的站点的请求转到 panel 这个 app 中，看这个 app 中的 urls.py 有没有对所访问的路径有进一步的指示：

# ss_panel/urls.py
from django.conf.urls import include, url
from django.contrib import admin

urlpatterns = [
    url(r'^admin/', admin.site.urls),
    url(r'', include('panel.urls'))
]

然后在 panel 这个 app 中新建一个空的 urls.py 在里面填写需要处理的 url 即可。

settings.py 中的 TIME_ZONE

TIME_ZONE 表示本地时区，修改为 TIME_ZONE = 'Asia/Shanghai' 即可。

login & logout

Django 可以使用 login 以及 logout 函数来完成用户的登录与登出操作。在写这一块儿的代码时出现了一些小的问题。

@login_required(redirect_field_name='')
def logout_view(request):
    logout_message = 'Goodbye, %s' % request.user
    logout(request)
    return HttpResponse(logout_message)

上一段代码是 logout 的操作。然而我在测试的时候（直接在浏览器输出 logout 的网址），会发现无论如何也不会输出 Goodbye 消息，后来折腾了一个小时，查看了 ./manager.py 的记录的时候发现，在浏览器输入网址的时候，浏览器会自动对这个网址进行一次访问，因此在这次访问时，用户就已经被注销掉了；而敲下回车时，会第二次对这个网址进行访问，此时用于早已经被注销，因此会被重定向到登录界面。后来直接加了一个链接，使用链接方式注销，问题解决。

Django Time Zone 问题

Django 在1.4版本之后存储如果设置了 USE_TZ=True，则存储到数据库中的时间永远是 UTC 时间。这时如果 settings 里面设置了 USE_TZ=True 与 TIME_ZONE = 'UTC'，用 datetime.datetime.now()获取的时间 django 会把这个时间当成 UTC 时间存储到数据库中去。如果修改设置为 USE_TZ=True 与 TIME_ZONE = 'Asia/Shanghai'，用 datetime.datetime.now() 获取的时间由于不带时区，django会把这个时间当成 Asia/Shanghai 时间，即东八区时间，然后 django 会把这个时间转成带时区 UTC 时间存储到数据库中去，而读的时候直接按 UTC 时间读出来，这就是网上很多人遇到的存储到数据库中的时间比本地时间会小8个小时的原因。

在所有需要时间的地方都使用 UTC 时间即可，在调试或者需要方便识别的地方如果需要使用本地时间，则将对应模型的 __str()__ 方法中打印时间的操作转化为打印本地时间就可以了。

Django 的分时间段统计

编写完 Log 模型后，我们可以得到每分钟的用户用量，然后可以使用某一种神奇的方式按小时（或者天、月）来统计每一个时段中用户的用量，具体代码如下。

Client.objects.extra(select={'day': connections[Client.objects.db].ops.date_trunc_sql('day', 'created_date')}).values('day').annotate(sum=Sum('account_balance'))
